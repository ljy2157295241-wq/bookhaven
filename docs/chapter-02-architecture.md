# 第2章 系统架构

## 2.1 总体架构设计

BookHaven 采用微服务架构，将系统拆分为 5 个独立的微服务和一个 API 网关。每个微服务围绕特定的业务领域构建，拥有独立的数据库、独立的部署单元，通过轻量级通信机制（HTTP + 消息队列）进行协作。

## 2.2 架构选型理由

| 组件 | 选型 | 理由 |
|---|---|---|
| **服务注册与发现** | Nacos | 相比 Eureka，Nacos 同时支持注册中心和配置中心，减少运维组件数量；支持 CP+AP 一致性切换 |
| **API 网关** | Spring Cloud Gateway | 基于 WebFlux 响应式编程，性能优于 Zuul 1.x；支持路由、过滤、限流等能力 |
| **服务调用** | OpenFeign + Resilience4j | OpenFeign 声明式 HTTP 客户端简化调用；Resilience4j 是 Spring Cloud 官方推荐的熔断组件（替代已停维的 Hystrix） |
| **分布式事务** | Seata AT 模式 | 对业务代码侵入小，自动解析 SQL 生成回滚 SQL，适合学生项目快速落地 |
| **消息队列** | RabbitMQ | 成熟稳定，支持可靠投递，适合订单支付异步解耦场景 |
| **缓存** | Redis | 缓存图书信息、JWT Token，减轻数据库压力 |

## 2.3 为什么采用微服务架构

1. **独立部署与扩展**: 下单高峰期可单独扩展 order-service，不影响其他服务
2. **故障隔离**: Product Service 宕机不影响用户登录和购物车
3. **技术异构灵活性**: 各服务可独立选择技术栈
4. **团队协作**: 虽然本次为单人项目，但微服务架构清晰地划分了模块边界

## 2.4 前端技术

- **Vue 3 + TypeScript**: 组合式 API + 类型安全
- **Element Plus**: 基于 Vue 3 的组件库
- **Pinia**: 替代 Vuex 的状态管理
- **Axios**: HTTP 请求库，对接 Gateway
